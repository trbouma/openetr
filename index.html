Durable Control. Portable Records.

A portable control layer for transferable records.

OpenETR is an open control layer for electronic transferable records: a Nostr wire format, installable Python component, CLI, and demonstration web app for records that need signed, inspectable control history outside any one platform.

The current work generalizes beyond a single document type. A record can be issued, transferred, encumbered, discharged, presented, and terminated, while legal or institutional recognition remains with the applicable framework.

Control Layer

Origin. Control. Recognition.

A small protocol surface for signed record evidence: origin events, control events, relay-backed state, and local recognition by the systems that rely on it.

  • Nostr kinds 31415 and 31416
  • Relay-backed profiles and aliases
  • Python, CLI, web, and protocol integration

Core Model

Objects, profiles, and events form one portable record layer.

Objects

The document or record identified by a digest and carried forward by an object id.

Profiles

Operational signers that issue, transfer, attest, encumber, discharge, or terminate records.

Events

Signed origin and control events that make the record history independently queryable.

Canonical View

The canonical model makes the scheme legible at a glance.

This diagram shows the basic OpenETR vocabulary in one frame: an object is the record surface, a controller is the party able to exercise control, and events are the signed transitions that move authority over time.

It is useful as both a conceptual map and an implementation guide, because it keeps the scheme focused on portable control rather than on any one platform, registry, or application boundary.

Current Implementation

The project now has a working control-event family and multiple integration surfaces.

General Control Events

kind 31415 introduces an object. kind 31416 records later actions such as transfer initiation, acceptance, attestation, encumbrance, discharge, redemption, and termination.

Root And Profiles

A root identity manages relay-backed configuration, while profiles act as operational signers for real workflow roles such as warehouse operator, exporter, bank, carrier, consignee, or attestor.

Queryable State

The query service derives origin, current controller, lifecycle state, control history, profiles, and outstanding encumbrances from signed relay events.

Why It Matters

Built for records that must outlive platforms and move with confidence.

Durable

Records persist independently of any single vendor, registry, or institution.

Portable

Control can move without re-issuance, lock-in, or proprietary dependencies.

Verifiable

Proofs are machine-checkable and can be validated outside the originating system.

Relay-Backed By Design

State can move between independent systems because the evidence is not trapped in one app.

Minimal Bootstrap

Only the root key and bootstrap relays need to be held locally or behind an account system. Profiles, aliases, defaults, and signer secrets are increasingly relay-backed.

Independent Relays

Origin and control events can be published and queried across relay infrastructure, allowing multiple systems to participate without sharing one database.

Independent Verification

Applications may cache, index, or render OpenETR state, but the signed protocol evidence remains portable and machine-checkable.

Integration Paths

Use the component, call a service, run the CLI, or publish the protocol directly.

OpenETR is meant to fit into existing systems without forcing every participant into one application boundary.

Installable Python Component

Existing Python services can import openetr and call reusable services for issuance, query, transfer, encumbrance, discharge, redemption, and termination.

Agent-Friendly CLI

The openetr command line is being shaped for shell workflows, scripts, CI jobs, and coding agents, with explicit inputs and parseable output modes as a design goal.

REST And Domain APIs

The FastAPI demonstration app exposes upload, issue, query, profile, and warehouse receipt routes that can be reused, wrapped, or replaced by production services.

Protocol-Level Integration

Integrators can publish and query Nostr events directly by following the OpenETR wire format for kinds, tags, object identity, and control-event traversal.

Account-Based Products

Existing platforms can hide root keys and bootstrap relays behind ordinary account flows while still preserving relay-backed OpenETR state underneath.

Independent Systems

Warehouses, banks, logistics platforms, registries, auditors, and relying parties can interoperate through shared event conventions rather than one shared backend.

Underlying Philosophy

Transact globally, validate locally.

OpenETR is designed for an open, permissionless environment: events can circulate globally, while validation, attestation, and recognition remain with the party that must decide whether to rely on the resulting chain.

Global Publication

OpenETR does not assume a single controlling platform or registry. It aims to let control-relevant events move through shared infrastructure while remaining signed, attributable, and reviewable.

Local Recognition

The protocol provides evidence. It does not by itself determine final legal or operational effect. That remains a matter for the assessor, attestor, or relying party applying the relevant policy.

Sound Cryptographic Basis

The goal is not to prevent every possible action at the protocol layer, but to create a sound cryptographic basis on which recognition decisions can later be made.

Historical Context

Inspired by the operating ethos of maritime trade on the high seas.

Trade Across Boundaries

Global shipping developed across ports, carriers, merchants, financiers, and legal systems long before any single sovereign could govern trade end to end.

Records That Travel

Maritime commerce depended on documents, customs, and recognition practices that could move across jurisdictions. Bills of lading mattered because they were portable records of control, not because they belonged to one system.

Digital Parallel

OpenETR is an attempt to replicate some of that ethos digitally: publication in a shared environment, movement across institutional boundaries, and recognition based on reliable evidence rather than centralized platform control.

Digital Trade Context

MLETR helps make digital trade documents legally usable across borders.

What It Is

The UNCITRAL Model Law on Electronic Transferable Records, or MLETR, provides a legal framework for electronic records to function like transferable paper instruments.

Why It Matters

For trade documents such as bills of lading, warehouse receipts, and promissory notes, MLETR creates a path for digital records to carry possession-like control and legal effect.

Why OpenETR Fits

OpenETR focuses on durable control, transfer, and independent verification, which are exactly the technical qualities needed to support MLETR-aligned digital trade documentation.

Warehouse Receipts

The current domain focus is the Model Law on Warehouse Receipts.

The MLWR Control Desk shows how a domain-specific application can speak warehouse receipt language while translating each action into the general OpenETR control model.

Domain Adapter

The web app exposes warehouse receipt actions: issue, query, transfer, accept, pledge or lien, release, presentation for delivery, and lifecycle completion.

General Core

Under the domain language, the app calls the same generic OpenETR services and publishes the same Nostr event families used by other record types.

Article Mapping

The specs now include an MLWR article requirements mapping that links legal concepts to implementation evidence and identifies remaining recognition-policy gaps.

Applicable Frameworks

One evidence layer can support multiple recognition frameworks.

OpenETR does not replace statutes, registries, courts, or private rulebooks. It supplies signed evidence that different recognition frameworks can evaluate.
